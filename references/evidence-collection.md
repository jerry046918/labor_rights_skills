# 证据收集与解析指引

> 本文件供 SKILL.md 在**阶段 3（证据收集与解析）**加载。
> 规定支持的证据类型、解析方式、统一卡片 schema、目录结构与可信度评级规则。

## 加载时机

阶段 2（动态深化）完成、用户已回答核心追问后加载。本阶段主动询问用户已持有的证据，并把每条证据解析为统一格式的卡片存入 `evidence/<session>/`。

---

## 一、支持的证据类型

| 类型 | 文件形式 | 解析方式 | 默认可信度 |
|------|---------|---------|----------|
| 文本 | 用户直接粘贴（聊天记录、邮件正文、通知书文字） | 直接进入上下文，抽取关键事实 | ★☆☆ |
| 图片 | `.png` / `.jpg` / `.jpeg`（截图、拍照） | Read 工具多模态视觉理解（需当前模型支持，否则见下方降级） | ★☆☆ / ★★☆（带公章原件） |
| PDF | `.pdf`（合同、工资条、银行流水、解除通知） | Read 工具读取 PDF（文本型 PDF 直接提取；扫描件需 OCR，见下方降级） | ★★☆（带公章）/ ★☆☆（无章） |
| 录音 | `.mp3` / `.wav` / `.m4a` | FunASR **本地**流水线（与对话所用 AI 模型无关，纯文本模型也能用） | ★☆☆ |

**不在本轮支持，但需要兜底处理：**

| 类型 | 兜底提示 |
|------|---------|
| Word / Excel | 提示用户："请复制粘贴关键内容到对话中，或导出为 PDF 后提供" |
| 视频 | 提示用户："本轮不支持视频解析。建议：① 用录音笔/手机单独录音；② 或导出视频中的关键聊天截图" |
| 纸质原件（未拍照） | 提示用户："请拍照并确保画面平整、光线均匀、文字清晰可读后提供" |

> ⚠️ **模型能力降级**：图片与扫描件 PDF 依赖当前对话模型的视觉/OCR 能力。若模型为纯文本（不支持视觉），或 Read 工具返回空/乱码，按下方**第八节"模型能力降级"**执行降级流程。**录音证据不受影响**——FunASR 是本地模型，与对话所用 AI 模型无关。

## 二、统一证据卡片 schema

**每条证据解析为一张卡片，存为单独 JSON 文件**：`evidence/<session>/cards/ev_XXX.json`

```json
{
  "card_id": "ev_001",
  "source_type": "text | image | pdf | audio",
  "source_path": "evidence/<session>/sources/<原文件名或文本文件名>",
  "source_filename": "解除劳动合同通知书.pdf",
  "provided_at": "2026-06-27T14:32:15",
  "parsed_summary": "一段话总结这条证据的整体内容（50-150 字）",
  "key_facts": [
    {
      "fact": "具体事实陈述（如：公司于 2026-05-15 书面通知解除）",
      "evidence_type": "解除意思表示",
      "dispute_relevance": "illegal_dismissal",
      "timestamp_in_source": "证据内的时间定位（如 '2026-05-15' / '第3页' / '00:03:12'）",
      "quote": "原文摘录（不超过 200 字，保留关键措辞）",
      "confidence": "high | medium | low"
    }
  ],
  "reliability_level": "★☆☆ | ★★☆",
  "speaker_role": "（仅录音）HR/人力资源 | 直属主管 | 员工本人 | ...",
  "cross_role_conflict": false,
  "cross_check_notes": "与其他证据的交叉校验结果（如 '与 ev_002 银行流水一致'）或 '未交叉验证'"
}
```

### 字段说明

| 字段 | 必填 | 说明 |
|------|------|------|
| `card_id` | 是 | 自增编号，三位数字起 `ev_001` |
| `source_type` | 是 | 枚举：text / image / pdf / audio |
| `source_path` | 是 | 相对仓库根的路径，指向 sources/ 下的原始文件 |
| `source_filename` | 是 | 原始文件名（保留用户提供的命名） |
| `provided_at` | 是 | 用户提供的本地时间，ISO 8601 |
| `parsed_summary` | 是 | 一段话总结证据内容，给意见书第四节引用 |
| `key_facts` | 是 | 抽取的关键事实数组（1-N 条），每条独立标注 |
| `key_facts[].evidence_type` | 是 | 见下方"证据类型分类" |
| `key_facts[].dispute_relevance` | 是 | 对应 dispute-patterns.md 的争议 token（如 `illegal_dismissal`） |
| `key_facts[].timestamp_in_source` | 否 | 证据内的时间锚点，便于回溯 |
| `key_facts[].quote` | 是 | 原文摘录，禁止改写 |
| `key_facts[].confidence` | 是 | agent 对该事实判断的置信度 |
| `reliability_level` | 是 | 见下方"可信度评级" |
| `speaker_role` | 仅录音 | 来自 role_mapping.json |
| `cross_role_conflict` | 仅录音 | 来自 conflict_detection.py 输出 |
| `cross_check_notes` | 否 | 跨证据交叉校验笔记 |

### 证据类型分类（evidence_type 枚举）

供 `key_facts[].evidence_type` 使用，按争议类型组织：

- **意思表示类**：解除意思表示 / 辞退通知 / 协商解除要约 / 离职申请 / 续签要约
- **合同类**：劳动合同条款 / 调岗条款 / 竞业条款 / 服务期条款 / 试用期条款
- **工资类**：工资标准 / 工资发放记录 / 工资扣减 / 工资条明细 / 银行流水
- **工时类**：考勤记录 / 加班安排 / 排班表 / 请假记录
- **社保类**：社保缴费记录 / 公积金缴费记录 / 放弃社保协议
- **工伤类**：诊断证明 / 工伤认定书 / 劳动能力鉴定 / 事故经过
- **沟通类**：微信聊天 / 邮件沟通 / 会议纪要 / 电话录音
- **管理类**：规章制度 / 员工手册 / 考核记录 / 处分决定
- **身份类**：工作证 / 名片 / 入职登记 / 录用通知

## 三、可信度评级

```
★★★  法律条文、行政法规、司法解释           （不在证据卡片评级范围）
★★☆  公文书证：带公章/签字的原件扫描、银行电子回单、官方平台 PDF（社保/个税）
★☆☆  视听资料、电子数据：聊天截图、录音、照片、用户口述、粘贴文本
```

**评级规则：**

1. **PDF/图片带公章** → ★★☆（公文书证）
2. **PDF/图片无公章**（聊天截图、内部通知截图、照片）→ ★☆☆（电子数据）
3. **银行流水 PDF** → ★★☆（银行业务公章或电子回单）
4. **个税 APP / 社保 APP 截图** → ★☆☆（截图可篡改，但可与官方平台核对）
5. **录音** → ★☆☆（统一，原因见 SKILL.md 阶段 3 子节）
6. **用户粘贴文本** → ★☆☆（无法溯源原始载体）
7. **微信聊天记录截图** → ★☆☆（建议提示用户保留原始手机载体）

**关键约束：** ★☆☆ 证据不可单独支撑关键结论。若某关键事实仅有 ★☆☆ 证据，意见书必须明确告知"该事实需进一步固定为 ★★☆ 证据"。

## 四、evidence/ 目录结构

```
evidence/
└── <session-id>/                          # 见下方命名规范
    ├── sources/                           # 原始文件副本
    │   ├── 解除劳动合同通知书.pdf
    │   ├── 工资银行流水.pdf
    │   ├── 微信聊天截图.png
    │   └── 谈判录音.mp3
    ├── cards/                             # 解析后的证据卡片（一文件一卡片）
    │   ├── ev_001.json
    │   ├── ev_002.json
    │   ├── ev_003.json
    │   └── ev_004.json
    ├── summary.md                         # 关键事实汇总（供意见书引用）
    │
    └── （仅当含录音证据时）
        ├── hotwords.json                  # 录音热词
        ├── result.json                    # FunASR 转写结果
        ├── role_mapping.json              # 说话人角色映射
        └── processing.log                 # 处理日志
```

### session-id 命名规范

格式：`<YYYY-MM-DD>-<化名>-<争议类型简称>`

示例：
- `2026-06-27-张某-违法解除`
- `2026-06-27-李某-拖欠工资`
- `2026-06-28-王某-加班费`

**规则：**
- 日期：用户首次提供证据的日期（不是争议发生日期）
- 化名：用户在阶段 1 提供的化名（如"张某"）；未提供时用"申请人"
- 争议类型简称：从 dispute-patterns.md 的 10 类选最核心的一个；多类并发时选主诉求
- 目录名禁用空格、`/`、`\`、`:`、`*`、`?`、`"`、`<`、`>`、`|`

### summary.md 格式

```markdown
# 证据解析汇总

**Session：** 2026-06-27-张某-违法解除
**解析时间：** 2026-06-27T14:45:00
**证据总数：** 4

## 关键事实清单

### 1. 劳动关系建立与终止
- 入职日期：2023-03-15【ev_001 劳动合同】★★☆
- 解除日期：2026-05-15【ev_002 解除通知书】★★☆
- 解除理由：业绩不达标【ev_002】★★☆
- 口头解除表述：2026-05-10 微信中"下周不用来了"【ev_003 微信截图】★☆☆

### 2. 工资标准
- 月平均工资：21,500 元【ev_004 银行流水】★★☆

### 3. 沟通记录
- HR 在 2026-05-10 谈话中提及"补偿 N"【ev_005 录音 00:03:12】★☆☆

## 交叉校验笔记
- ev_002（解除通知载明 2026-05-15）与 ev_003（微信 2026-05-10 提及解除）时间顺序一致 ✓
- ev_004 银行流水月均 21,500 元与用户口述"月薪 2 万"基本一致 ✓

## 未交叉验证的事实
- 加班事实仅有用户口述，建议补充考勤记录

## 缺失的关键证据
- 前 12 个月工资流水（用户仅提供 6 个月）
- 工会沟通记录（如涉及 §43 程序审查）
```

## 五、解析流程（agent 执行步骤）

### Step 1：主动询问证据

阶段 3 启动时，agent 主动询问：

> 您手里目前有哪些证据？可以提供以下任一形式：
> - **文字**：直接粘贴聊天记录、邮件、通知书内容
> - **图片**：截图、拍照（合同、工资条、通知书、微信聊天等）
> - **PDF**：合同、银行流水、解除通知等
> - **录音**：mp3/wav 文件路径（谈判、电话录音）
>
> 没准备好的话，可以一项一项慢慢来，也可以稍后再补充。

### Step 2：创建 session 目录

收到第一条证据时：
1. 用 session-id 命名规范创建 `evidence/<session-id>/` 及 `sources/`、`cards/` 子目录
2. 后续所有证据归入同一 session

### Step 3：逐条解析

对每条证据：

**3a. 文本证据**
- 直接进入 agent 上下文
- 抽取关键事实，标注 `evidence_type` 与 `dispute_relevance`
- 写入卡片文件
- 同时把原文存为 `sources/ev_XXX.txt` 备查

**3b. 图片证据**
- Read 工具读图（多模态视觉，需当前模型支持；纯文本模型走降级路径见下方第八节）
- 提取文字内容、识别公章/签名、判断截图来源（微信/钉钉/邮件）
- 抽取关键事实，写入卡片
- 原图复制（或路径引用）到 `sources/`

**3c. PDF 证据**
- Read 工具读 PDF（已支持）
- 提取关键条款、金额、日期、公章信息
- 抽取关键事实，写入卡片
- 原文件路径存入 `sources/`（PDF 通常较大，可不复制只引用路径）

**3d. 录音证据**
- 触发录音专门处理流水线（见 SKILL.md 阶段 3 子节"录音证据的专门处理"）
- 转写 + 角色推断 + 证据卡片抽取
- 录音卡片复用本统一 schema，`source_type: "audio"`，附带 `speaker_role` 与 `cross_role_conflict` 字段

### Step 4：交叉校验

所有证据解析完成后：
- 同一事实被多条证据支撑 → 在 summary.md 标注交叉校验通过 ✓
- 同一事实出现矛盾 → 在 summary.md 标注矛盾点，提示用户核实
- 关键事实仅有 ★☆☆ 证据 → 在 summary.md "缺失的关键证据"章节提示固定建议

### Step 5：生成 summary.md

按上方格式汇总，作为阶段 5（意见书生成）第四节的引用源。

### Step 6：告知用户

告知用户：
- 已解析的证据数量
- 关键事实清单（用直白语言）
- 建议补充的证据清单（从 dispute-patterns.md 该类型的"证据清单"对照）

## 六、隐私处理

**所有证据类型通用：**
- 不收集身份证号、银行账号、家庭住址等敏感信息
- 如证据中出现敏感信息，agent 主动提示："检测到您提供的证据中包含身份证号/银行账号，建议在意见书中以化名/部分隐藏方式引用"
- 公司名称可用化名（不影响法律分析）
- 所有证据文件保存在本地 `evidence/` 目录，不上传

**录音特殊处理：**
- 录音转写后的文字会进入当前 AI 模型上下文
- 详细隐私说明见 SKILL.md 阶段 3 子节

---

## 七、与 dispute-patterns.md 的衔接

阶段 3 完成后，agent 应：

1. 对照 `dispute-patterns.md` 该争议类型的"证据清单"
2. 比对用户已提供的证据 vs 清单中的"必备/重要"项
3. 在 summary.md "缺失的关键证据"章节列出差距
4. 这些差距直接驱动阶段 5（意见书）第四节的"待固定证据建议"

## 八、模型能力降级（纯文本模型场景）

> **触发条件**：用户使用纯文本模型（不支持视觉），或 Read 工具读取图片/扫描件 PDF 时返回空、报错、乱码。

### 8.1 哪些证据受影响

| 类型 | 是否受影响 | 原因 |
|------|----------|------|
| 文本（用户粘贴） | ❌ 不受影响 | 直接进入对话上下文 |
| 图片（截图/拍照） | ✅ 受影响 | 需要 Read 多模态视觉能力 |
| PDF（文本型，可复制选中文字） | ✅ 部分受影响 | Read 通常能提取文本，但扫描件 PDF 不行 |
| PDF（扫描件，图片型） | ✅ 受影响 | 需要 OCR 能力 |
| 录音 | ❌ 不受影响 | FunASR 完全本地，与对话模型无关 |

### 8.2 降级触发判定

agent **不应该预先假设**自己是不是多模态——只能通过尝试 + 失败来判断：

1. **先正常尝试 Read**：用户提供图片/PDF 时，先用 Read 工具读取
2. **判断结果**：
   - ✅ 成功（提取到有意义文字/视觉理解返回正常描述）→ 按正常流程解析
   - ❌ 失败信号：
     - 工具调用报错（"model does not support vision" 或类似）
     - 返回空字符串或纯乱码
     - 返回的内容明显与图片无关（幻觉/占位文本）
3. **触发降级**：明确告知用户当前能力局限，给出可选路径

### 8.3 降级路径（按推荐顺序）

agent 在降级时应向用户呈现以下选项：

```
⚠️ 当前模型可能不支持图片/扫描件直接读取。以下几种方式可以继续：

【选项 A】（推荐，最快）您把图片/PDF 中的关键内容用文字描述给我
   例：解除通知书写明"2026-05-15 因业绩不达标解除"
   我会按文字证据处理（可信度 ★☆☆，无法溯源原件）

【选项 B】用本地 OCR 工具转成文本后粘贴
   Windows 11 自带：截图工具 → "文本操作"按钮
   WPS：打开图片 → "图片转文字"
   Adobe Acrobat：扫描件 PDF → "增强扫描"
   微信："提取文字"功能
   处理后把文字粘贴给我

【选项 C】换用支持视觉的 AI 模型
   如 GPT-4V / Gemini Pro Vision / 其他多模态大模型
   切换后重新提供该图片

【选项 D】（仅扫描件 PDF）转为文本型 PDF
   用 OCR 工具处理后另存为 PDF
   再提供处理后的 PDF
```

### 8.4 降级卡片标注

走降级路径（A 或 B）后写卡片时，必须明确标注降级路径：

```json
{
  "card_id": "ev_003",
  "source_type": "text",
  "source_path": "evidence/<session>/sources/ev_003.txt",
  "provided_at": "...",
  "parsed_summary": "（用户口述/OCR 转写）解除通知书记载于 2026-05-15 解除",
  "key_facts": [...],
  "reliability_level": "★☆☆",
  "degradation_note": "原文为图片，当前模型不支持视觉；用户口述关键内容（选项 A）",
  "original_source_kept": "用户本地保留原图 ev_003_source.png"
}
```

新增字段：
- `degradation_note`：标注降级原因与路径（仅降级时填写）
- `original_source_kept`：提示原始证据在用户本地（agent 不可见，仅在仲裁/诉讼时由用户提交原件）

### 8.5 录音证据的独立性

**强调**：录音证据处理与对话所用 AI 模型完全无关：
- ASR 转写：本机 FunASR 模型（不依赖对话模型）
- 角色推断：在转写文字上做（纯文本任务，任何模型都能处理）
- 卡片抽取：在转写文字上做（纯文本任务）

因此**即使用户使用纯文本模型，录音证据流水线仍可完整执行**。仅最后一步"接入意见书"时受模型能力影响（与其他证据一致）。

### 8.6 阶段 3 启动时的预告知（可选）

若 agent 通过某种方式（如环境变量、用户配置、首次 Read 失败）已知当前为纯文本模型，可在阶段 3 启动询问证据时主动加一句：

> 💡 提示：当前对话模型似乎不支持图片读取。如果您有截图/PDF 证据，可以：
> ① 用文字描述关键内容粘贴给我；
> ② 或换用支持视觉的模型后重新启动。

否则不主动声明，保持简洁。
